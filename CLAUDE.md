## 1. 開発方針（最重要・維持すること）

### 1-1. 裏取り（今日いちばん徹底した原則）
- **本番に入れる前に、必ずサンドボックスで実データ（実 `db.json`・実図面）で検証する。**
- **図面が絡む判断は「Vision で読んで裏を取ってから」直す。** 決めつけない。
  - 由来：初回、コンデンサ盤2系統を裏取りせず「重複バグ」と決めつけかけた反省から。
  - 実践例：タイル分割の逆効果も、制御盤の「分岐回路コード方式」も、実図面を Vision で読んで確認してから結論を出した。

### 1-2. 副作用チェック
- 改修が**他機器を壊さないか**を毎回確認する。目安は「同一入力で ◎/○/△ が変わらない（回帰ゼロ）」を A/B で実測。

### 1-3. 選定の安全側原則
- **◎（確定）の誤答ゼロを最優先。** ◎をむやみに増やさない。**迷ったら △（安全側）。**
- **AI はコードを生成しない。** DB 実在コードのみを選定する。情報不足は △。
- 抽出ミスは積算の土台を崩す致命的ミス。**全機器・全盤を等しく精査**（注目機器だけ見て流さない）。

### 1-4. セキュリティ
- 認証情報（API トークン・SSH 秘密鍵）を**画面・ログ・チャットに出さない。** 露出したら必ず**失効（Revoke）**する。
- `Estimate_system0630` の `*.pem`（SSH秘密鍵）は共有範囲に含めない。安全な場所へ退避推奨。

### 1-5. 再現性
- `temperature`/`top_p`/`top_k` は Opus 4.7+ で非デフォルト値だと 400 エラー。**指定しない。**
- 同一図面には同一結果を返す一貫判断。画像 SHA256 キャッシュ（`_EXTRACT_CACHE`、再起動で消える）。

## 2. コスト削減の方針と現状（本セッションで新規検討）

**大前提：精度（◎誤答ゼロ）を犠牲にしないコスト削減。** 安いモデルは「安全に読める前提の一次段」に限定し、難所は上位モデルへ。

コスト構造：**費用はほぼ抽出（Vision）に集中**。選定は DB照合＋ルールで約1秒＝ほぼ無料。

### 検討した施策と結論（正直な実測込み）
| 施策 | 状態 | 知見 |
|---|---|---|
| **プロンプトキャッシュ** | 配線可・害なし | `EXTRACT_PROMPT` が約2〜3kトークンでキャッシュ最小(4,096)未満→**現状は効果限定**。プロンプト拡張時に自動で効く。 |
| **モデル階層（Haiku一次→Opus難所）** | 有望 | Haikuは入力$1/出力$5（Opusの1/5）。ただし**タイルで読める前提**（HaikuはA3全体では精度不足）。 |
| **タイル分割（`TILE_GRID`）** | ⚠️**実験的・既定OFF維持** | 検証したのは「タイル＋**Haiku**＋名称マージ」で、密な図面（電灯分電盤）で**逆効果を実測**：盤断片化(10→33〜47面)、確定率 90%→42〜59%。ただし**失敗要因は概念でなく①安価モデル②素朴マージ**。「タイル＋**Opus**（精度重視）」は**未検証**（＝Geminiが勧める本命）。 |
| **品質ゲート** | 有効 | 読取不明瞭が高い図面は攻めず「より精度の良い図面（元CAD由来PDF等）」を要求。コスト・精度の両面で効く。 |
| **バッチAPI（50%引き）** | 候補・未実装 | 非同期（夜間一括）で許容できる用途向け。 |
| **他プロバイダ（Gemini/DeepSeek）** | 後回し候補 | 一次段の差し替え候補だが、プロンプト/JSON整合・保守コストが増える。**まず自社内（Haiku＋キャッシュ）で稼ぐ**のが安全。 |

> 教訓：**「機器検出数が増える＝良い」ではない。** タイル分割は検出数を増やしたが、断片化・低品質化で確定率を下げた。数だけでなく**確定率と安定性**で評価する。

> タイル分割は**目的で分岐する**：〔精度重視〕タイル＋上位モデル(Opus/Gemini Pro)＋良いマージ＝誤読ほぼゼロだがコスト増／〔コスト重視〕タイル＋Haiku＝安いが今回は精度低下。外部所見（Gemini 2.5 Pro に本図面を読ませた報告）でも「1ページに結線図＋器具表がある図面は4〜6分割して上位モデルで個別に読む」ことが推奨されており、**精度重視のタイル＋上位モデルは有望**。未検証なので次に「タイル＋Opus＋改良マージ vs 1x1」を A/B する価値がある。なお細い**ケーブルサイズ**文字の誤読は**見積対象外**（対象は盤類）なので積算額に影響しない。

5. ロールバックは各スクリプトが表示する `cp *.bak_日時 ...`。

## 4.5 二重Vision＋裁定（確実な抽出／本セッションで実証）

**目的：◎の誤答ゼロ。** 抽出を2つの独立したVisionで読み、突き合わせる。

```
Claude(1回目) ∩ Gemini 2.5 Pro(2回目)
  ├ 一致            → ◎（二重検証済み・信頼）
  └ 不一致          → Claude が該当箇所を精査して裁定
                       ├ 判別できた → 採用（○）
                       └ 判別不能   → △（人が確認）
```

- **2回目は Gemini が安い＆強い**：Gemini 2.5 Pro は入力 約$1.25/出力 約$10（Opus $5/$25 の 約1/4・2/5）。しかも別モデルなので独立検証になり、Claude 2回より誤読を捕まえる。
- **実測（受変電 amagasaki.pdf）**：◎60 / ○27 / **△9（真に曖昧な機器のみ／負荷明細は対象外除外済）**。誤った◎ゼロ。突合が表記違いでペアできない機器も**黙って落とさず△**＝安全に失敗する。
- **突合の要点**：名称正規化（機器コード集合＋数値集合の一致）、別名辞書（`TR=T`, `SRX=SR`, `VMC=VCS` 等）、**負荷明細（L-/M-/EV/直圧/PU/消火栓P 等）は突合前に対象外除外**。
- **実装**：Gemini は Google AI Studio の APIキー（新形式 `AQ.` で始まる。旧知識の `AIza` は誤り）、`google-genai` SDK、`model='gemini-2.5-pro'`、`response_mime_type='application/json'`。キーはリポジトリ外（`.gemini_key`、gitignore）に保持。**使用後は AI Studio で失効**。

### コード選定は決定的（Claudeは悩まない）／悩む時はGeminiに“ルールごと”聞く
- 選定は **DB照合＋ルールで約1秒・決定的**（AIはコード生成しない）。通常は悩む余地なし。
- **悩む＝`ai_pick`（DB候補からLLMが1つ選ぶ）に落ちた曖昧時のみ**。ここだけ **Claude と Gemini が同じDB候補から選び→一致で採用／不一致は△（人）**。抽出と同じクロスチェックを選定にも適用。
- **Gemini に第2意見を求める際は、DB候補だけでなく「選定ルール（ドメインルール）」も必ず一緒に渡す。** 素の推測をさせず、御社慣行に沿わせる。共有すべき主なルール：
  - 高圧コンデンサ盤の SR/SC は系統が高圧＝**45系**（234V等の低圧表記に惑わされない）。SRX=SR、VMC→VCS解釈（要確認△）。
  - 高圧 TR/SC/SR は購入品コードが無いので**支給品系（45系）を既定**。容量は仕様以上で最近傍上位（切上げ）。
  - MDA（マルチ指示計 42075-42088）＝上位で出たら案件全体で計器を統一（型式要確認△）。TR/SC/SR/MCB等の機器名を含むものは継承対象外。
  - 動力制御盤は**分岐回路コード（22-29系）**：主回路パターン→回路種別、kW→容量ステップ切上、●=MCCB/○=ELB。
  - **分岐MCB/ELBの端子台ルール（社内仕様・単価で裏取り済）**：同じMCB/ELBでも盤種で端子台の有無が変わり別コード・別単価。
    - **分電盤＝端子台なし（60系）**：客先外線を盤内機器へ直接接続。例 3P100A→60133。
    - **動力制御盤＝端子台付き（50系）**：端子台を設け外線→端子台→盤内配線でMCBへ。端子台分だけ高い。例 3P100A→50133（50系>60系を単価で確認：50133=34>60133=32、50522=12>60522=10）。
    - **受変電の低圧電灯盤/低圧動力盤＝40系**（例 3P100AF→40133、AX付は41系、スペース/予備は末尾93）。
    - ∴ **盤種判定（制御盤/分電盤/受変電低圧盤）が端子台=系統を決める**。盤種を取り違えると端子台も誤るので分類は要精査。
  - **配電盤セットの計器種別・VCB操作方式は図面に明記されないことが多い**（特記仕様/客先打合せで決定）。
    抽出後・選定前の確認ゲートで人が確定（既定=普通角/手動のまま◎にしない＝○止め）。実見積書で八戸=マルチ・東部=広角と案件差を確認。
  - ケーブル類・計器ヒューズF・負荷明細は計上対象外（PFは計上）。ZCTはLGR選定時に親吸収。DS×3は三相=3P1台。
  - **迷ったら△（安全側）。◎はむやみに増やさない。**

## 6. 配電盤セットコード方式（本セッションで確立／`experiments/setcode/`）

人の見積は「部品を1点ずつ拾う」のではなく、**盤種＋数個の仕様でセットコードを1つ選ぶ→積算ソフトが部品一式へ展開・積算**する。本システムの役割は**抽出→セットコード選定まで**。展開・単価は積算ソフト（別ソフト・単価DBもそこ）の管轄で、**本システムでは単価を気にしない。出力＝コードの列**。

### 6-1. 判明した重大欠落（積算コード表↔db.json 全52ページ照合）
- **配電盤セットコード（11高圧/16段積/17低圧/19JEM）がdb.jsonに丸ごと欠落**（＝木村病院で配電盤0〜19%だった根本原因）。制御盤分岐(22-29)・非常用(30)のセットは実装済みだった。
- 個別部品(40-98系)は名称・単価ほぼ完全一致。欠落＝配電盤セット122＋2P用22＋個別31(90系板金15・97系認定4・40系4P5等)。詳細 `experiments/setcode/db_reconcile_findings.md`。

### 6-2. セットコード体系（積算コード表 p3索引 / p7-9・p25 内容）
- 11=高圧セット(受電/饋電) 16=段積 17=低圧セット(TR支給・計器まわり) / 22-29=制御盤分岐(実装済)。
- **17系は全てTR支給前提**。p7とp8は対で、p8「配電盤セットコード内容」がp7コードを図解。**引出型＝VCB上下の「黒四角＋回転矢印」記号**（手動/電動と区別＝VCBコード・OCR型・11系/16系の別が決まる）。
- 段積の組立：受電盤=単独1面／二段積=一段積×2 or 一段積+母連／三段積=一段積×3 or 一段積×2+母連。1ヶSP=1段スペース(機器なし)。11系電動=手動+CS×1+AUX-RY×4(p8脚注)。

### 6-3. 選定パイプライン（`experiments/setcode/setcode_selector.py`）
```
図面 → 抽出(盤属性) → 【確認ゲート】読切れない仕様を人に問う → セレクタ(決定的) → セットコード
```
- **選定は決定的**（選定属性 settype/meter/phase/role/vcb/op/cap の完全一致＋容量は仕様以上で最近傍上位＝切上）。AIはコード生成しない。
- **確認ゲート**：抽出で確信を持って読めた仕様のみ採用。読切れない仕様は**推測せず選定前に人へ確認**。**UIはコンボボックスで選択肢提示・自由入力させない**。
  - **計器種別(meter)は `ALWAYS_CONFIRM`＝常に確認ゲートに載せる**（単線図で確定しづらく誤ると◎誤答になるため）。抽出値があれば初期選択、無ければ**既定=普通角**。実API実測で計器種別は確定困難と確認され、常時確認が妥当と裏付け。迷ったら△。
- 実証（**本物の見積書ベース**／`validation_results.md`）：とくなが病院・船引・東部給食・八戸酒造の**人の見積書**で低圧TR **20/20** が候補DBに命中。展開も見積書明細と核一致。
  - ⚠️ 旧 `kimura_answer.json` は**別チャットのAIが拾ったもので正解ではない**（メータ種別を誤記＝マルチ等）。木村「7/7」はAI答えへの自己一致で真値ではない。正解は**人の見積書**。
  - 計器種別は案件で分かれる（とくなが/東部=広角、船引/八戸=マルチ）→ **確認ゲート必須**の裏付け。
- 実API実測（木村単線図E-004）：拡張プロンプトで抽出→セレクタ。**低圧TRの相・容量を正しく読みセット◎**（17206/17205/17238/17236/17233）。受電/饋電は計器種別を推測せずunclear→確認ゲート作動、スコットは容量unclear→確認。**"図面→抽出→確認ゲート→セレクタ"が実機で機能**。

## 7. 盤付属品の艤装ルールと外形図（本セッションで茂泉様確定・実装済）

単線図に描かれない盤の艤装品（換気扇/照明/ヒータ）は、盤種別の艤装ルールで自動計上（○）し、確認ゲートで人が員数・種別を調整（◎上書き）。実見積書7案件の照合で確立。

### 7-1. 艤装ルール（`_std_accessories` / `_panel_fan_code`）
- **照明**＝配電盤(受変電=haiden)一面に `71091`(FL-10WﾄﾞｱSW付) を1つ。**制御盤・分電盤には付けない**（実測: 照明は全案件71091、配電盤面あたり約1個）。
- **換気扇**＝①**まず外形図に描かれていればその通り拾う**（配電盤=屋外型`74102`/制御盤=制御盤用`74103`。負荷の換気扇 FE/FS 3φkW・遮断器付は除外し分岐遮断器で拾う）。②外形図に無ければ **TR≥500kVA が入る配電盤面** に`74102`を1つ（①がある面は二重計上しない=排他）。

### 7-2. 外形図（機器外形図/正面配列図）から面数・寸法・段数（`outline`）
- 抽出で配電盤に `outline:{faces,w,h,d,stories,sections:[{name,tr_kva,fan}]}` を付与。
- **面数＝照明員数の正解値**（実証: 六本木 面数11=見積書照明11／八戸 面数7=見積書照明7 と完全一致）。外形図があれば配電盤の照明=面数・換気扇=外形図の換気扇数を**一括計上し、電気盤(単線図)ごとの自動計上を抑制**（電気盤数>物理面数の過剰計上を解消。六本木 照明26→11）。
